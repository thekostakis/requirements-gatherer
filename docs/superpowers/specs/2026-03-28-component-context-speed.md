# Component Context Speed Optimization Spec

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Speed up the component-context skill by introducing a component index file and dispatching a Haiku subagent for spec lookup instead of doing the work in main context.

**Problem:** The component-context skill runs slowly because it operates in main context (Opus) and, for fuzzy matching, reads every spec file in `design/components/` individually. A design system with 20+ components means 20+ Read calls on an expensive model.

---

## 1. Component Index File

**Path:** `design/components/index.md`

A lightweight manifest summarizing all component specs. Generated by the consultant during wrap-up and kept current by a dedicated update skill.

### Format

```markdown
# Component Index

| Component | Category | Related | Keywords |
|-----------|----------|---------|----------|
| alert | feedback | toast, badge, form-error | status, warning, confirmation, dismissible |
| button | action | link, icon-button | click, submit, primary, secondary, disabled |
| card | layout | list-item, tile | container, content, image, clickable |
| modal | overlay | dialog, drawer | popup, close, backdrop, form |
```

### Field Definitions

- **Component:** Filename without `.md` extension (matches `design/components/[name].md`)
- **Category:** Inferred from the spec's Usage & Content section. One of: feedback, action, navigation, layout, overlay, data-display, form, media. If unclear, use the closest match.
- **Related:** Comma-separated list from the spec's "Related components" field. Empty string if none.
- **Keywords:** Comma-separated, extracted from the spec's "When to use" and "Common contexts" fields. Cap at 8 keywords per component. Pick distinctive words, skip generic ones (the, a, for, when, used).

### Generation Rules

- One row per `.md` file in `design/components/` (excluding `index.md` itself)
- Sorted alphabetically by component name
- If a spec lacks a Usage & Content section, set Category to "uncategorized" and Keywords to the component name only
- If a spec lacks a Related components field, set Related to empty

## 2. Rewritten Component-Context Skill

**Path:** `plugins/visual-design/skills/component-context/SKILL.md`

The skill becomes a thin dispatcher that identifies the target component and launches a Haiku subagent to do the lookup.

### Skill Flow

1. **Identify** the component from context (current task, file being edited, user request).
2. **Dispatch** a Haiku subagent via the Agent tool with:
   - The component name to look up
   - The full lookup instructions (inline in the SKILL.md as a prompt template)
3. **Present** the subagent's result directly in conversation.

### Subagent Prompt Template

The SKILL.md contains the subagent prompt as a template string. The skill substitutes `{COMPONENT_NAME}` before dispatch. The subagent instructions include:

**Exact match path:**
1. Try `design/components/{COMPONENT_NAME}.md` directly.
2. Try variants: hyphenated, camelCase-split, singular/plural.
3. If found, read the full spec. Skip to token loading.

**Fuzzy match path (no exact match):**
1. Read `design/components/index.md`.
2. Score each row against the target using:

| Signal | Weight | How it matches |
|--------|--------|---------------|
| Name substring | 3 | Target name appears in Component column or vice versa |
| Related components | 3 | Target name appears in Related column |
| Keyword overlap | 2 | Target name or context words appear in Keywords column |
| Same category | 1 | Target's likely category matches the row's Category |

3. Read the full spec files for the top 3 scoring rows (score > 0).
4. Apply the existing delivery format:
   - Suggestion 1: full spec
   - Suggestions 2-3: abbreviated (Visual Properties, Data Fields, Usage & Content only)

**Token loading:**
- Read `design-guidelines.md` and extract tokens referenced by the spec (colors, spacing, typography, motion, breakpoints).

**Delivery format:**
- Exact match: `## Design Spec: [Name]` + full spec + `## Relevant Design Tokens` + tokens + optional `## Responsive Tokens`
- Fuzzy match: `## No Exact Spec: [Name]` + ranked suggestions + consultant skill suggestion
- No match: "No spec file found for [name], and no similar components in the compendium. Run the visual-design-consultant skill to add it."

### What Moves Into the Subagent Prompt

- All fuzzy scoring logic (currently in SKILL.md Steps 3-6)
- Delivery format templates
- Context limits (3 specs max, 50 lines tokens, abbreviation rules for suggestions 2-3)

### What Stays in the Skill

- YAML frontmatter with trigger phrases (unchanged)
- Component identification logic (Step 1)
- Agent dispatch call
- A note to present the result directly

### Trigger Phrases (unchanged)

"load component spec", "component design spec", "what does the design say about [component]", "show me the spec for [component]", "design spec for [component]"

## 3. New Skill: Update Component Index

**Path:** `plugins/visual-design/skills/update-component-index/SKILL.md`

Regenerates `design/components/index.md` from the current spec files. Use when specs are manually added, edited, or removed.

### Trigger Phrases

"update component index", "rebuild component index", "refresh design index", "regenerate component index"

### Workflow

1. Check that `design/components/` exists. If not, STOP: "No component compendium found."
2. Glob all `.md` files in `design/components/` excluding `index.md`.
3. For each spec file:
   a. Read the file.
   b. Extract component name (filename without `.md`).
   c. Extract category from Usage & Content section (or "uncategorized").
   d. Extract Related components (or empty).
   e. Extract keywords from "When to use" and "Common contexts" (cap at 8).
4. Sort rows alphabetically by component name.
5. Write `design/components/index.md` with the standard format.
6. Compare against previous index (if it existed):
   - Report added components
   - Report removed components
   - Report components with changed metadata
7. Tell the user what changed.

### Constraints

- Never modify spec files -- read only
- If `index.md` doesn't exist yet, create it (first run after manual spec creation)
- If `design/components/` has no `.md` files besides `index.md`, write an empty index with just the header

## 6. Performance Impact

**Before:** Component-context reads N spec files (all on Opus). For a 20-component design system with no exact match, that's ~20 Read calls + scoring in Opus context.

**After:** Main context does 1 Agent dispatch call. Haiku subagent reads 1 index file + 1-3 full spec files + design-guidelines.md. Total: ~5 Read calls on Haiku.

Speed improvement comes from two factors:
1. Haiku is faster than Opus for this lookup task
2. The index eliminates reading N files -- the subagent reads 1 index to narrow candidates, then reads only the top 3

---

## 7. Constraints

- Index file must stay under 100 rows (one per component)
- Keywords capped at 8 per component
- Subagent reads at most 3 full spec files for fuzzy matching
- Subagent returns at most 3 suggestions
- Index is regenerated (not appended) on every update -- always reflects current state
